SEC 指标系统 Databricks 生产化方案

供产品经理、数据工程与前端共同评审的每日更新、版本管理、数据存储和 API 输出方案。

Jul 13, 2026, 9:08 AM UTC

Executive Summary

  • Databricks 可以承载完整数据链路，但要区分存储、计算与应用服务。 Delta 表是权威数据存储，SQL Warehouse 是查询计算资源；Lakebase 是可选的低延迟应用数据库，不应被三者混称为“数据库”。
  • 第一版保持很小。 使用一条每日 Lakeflow Job、一个 Unity Catalog、Bronze/Silver/Gold 三层 Delta 表、一个只读后端 API；不引入 Airflow、Kafka、复杂 agent harness、Lakeflow Declarative Pipelines 或预览能力作为关键依赖。
  • 版本历史必须显式保存。 每次通过验证且内容发生变化时生成一个 dataset_version，并保存该版本的完整 metric_snapshot。Delta Time Travel 只用于运维排错和短期恢复，不承担产品级历史查询。
  • AI 不进入指标主链路。 当前指标值、状态、证据和 notes 均由确定性规则生成。若未来使用 Claude API，只能写入隔离的辅助注释表，不能改变指标值、状态或发布结果。

权威数据留在 Delta，网站通过 API 读取

推荐链路是：SEC 官方接口 → Unity Catalog Volume 原始证据 → Bronze 元数据 → 现有确定性计算 → Silver 校验 → Gold 版本快照 → SQL Warehouse → 只读 API → 前端。

Lakeflow Jobs 已提供调度、任务依赖、条件分支、重试和通知，因此没有必要再建设一套编排 harness。Unity Catalog Volumes 用来保存原始 JSON、HTML、inline XBRL 文件和响应元数据；Delta 表只保存可查询的清单、标准化事实和发布数据。

关键边界：浏览器绝不能直接连接 SQL Warehouse。 前端不持有 Databricks token，不提交任意 SQL，也不感知 warehouse id。后端 API 使用独立 service principal，只拥有 Gold 视图的 SELECT 权限。

每日运行先发现变化，再计算、验证和发布

系统仍然是 company fact / accession evidence 驱动，不是 Daily Index 驱动。每天先读取配置中的 CIK，再通过 SEC Submissions API 发现与任务相关的新 accession 或 amendment；需要数字事实时更新 Company Facts，需要特殊指标时抓取对应 10-K、10-K/A、DEF 14A、8-K 或附件。

外部下载只获取有变化的证据，但计算阶段可以从缓存证据重算全部十家公司。当前数据量很小，这比增量拼接更容易证明正确，也更符合已有全量交叉检查。没有相关变化时，任务记录 NO_CHANGE 后结束，不创建空版本。

推荐建立一个生产 catalog，例如 sec_metrics_prod，其中包含 bronze、silver、gold 三个 schema，以及 bronze.sec_raw Volume。原始文件以 SHA-256 内容寻址保存，重复下载不会产生重复文件。

最重要的 Gold 对象是 dataset_version、metric_snapshot 和 metric_evidence。metric_snapshot 保存每个已发布版本的完整快照，而不是只保存变化行；以当前规模，这种设计的存储成本很低，却显著降低前端查询、回滚和审计复杂度。

业务版本是永久数据，不是 Delta 的临时历史

是的，必须保存“2026 年第一季度的某指标在 7 月 10 日是 100、在 7 月 13 日变成 103”这两个观察版本。每条记录同时具有三类时间：

  • period_start / period_end：指标代表的业务期间；
  • source_filed_at / source_accepted_at：SEC 来源何时提交；
  • observed_at_utc / published_at_utc：本系统何时看到并对外发布。

Databricks 明确说明，表历史和 Time Travel 会受保留期及 VACUUM 影响，也不建议作为长期归档。因此产品历史必须保存在显式版本表中；Time Travel 只作为短期运维工具。官方说明

发布标记最后写入，避免前端看到半成品

一次发布按以下顺序执行：先生成新的 dataset_version_id；写入该版本的完整 metric_snapshot 和 metric_evidence；执行唯一性、证据、状态、差异和既有 Golden/Validation 检查；最后才把 dataset_version 写为 PUBLISHED。所有 Gold 查询只读取 PUBLISHED 版本。

Databricks 已提供跨表事务能力，但当前仍属于 Public Preview。第一版不把关键发布一致性建立在预览功能上，而采用“发布标记最后写入”的稳定协议。Databricks ACID 与事务说明

发布后的数据不更新、不删除。修正或回滚都创建新版本，并记录 previous_version_id、change_type、change_reason、code_version、config_hash 和 dataset_hash。

前端先固定版本，页面内所有请求都使用同一版本

页面加载时先调用 GET /api/v1/datasets/current，获得当前 dataset_version_id。随后该页面的指标、证据和历史请求都显式携带这个版本。即使后台在用户浏览期间发布了新版本，页面也不会出现一半旧值、一半新值。

API 返回值必须包含：dataset_version_id、公司、指标、期间、value_numeric 或 value_text、单位、业务状态、质量状态、来源 accession/form/filing date、证据数量、发布时间和变化原因。数值建议以十进制定点字符串输出，前端负责格式化，避免 JavaScript 浮点误差。

第一版用 SQL Warehouse；只有严格低延迟场景才加 Lakebase

对于内网、十家公司和中等并发，推荐 Databricks App 或现有内网后端 + Serverless SQL Warehouse + 按版本缓存。Gold 层提供已经扁平化的查询表或普通视图，API 不在请求时进行复杂联表。Databricks App 可通过专用 service principal 访问 SQL Warehouse，避免硬编码个人 token。官方说明

如果将来变成公开网站、持续高 QPS 或要求稳定亚秒级响应，再把“当前已发布快照”同步到 Lakebase 作为 serving copy。Delta Gold 仍是唯一事实来源，Lakebase 只是可重建缓存。第一版不需要同时维护两套数据库。

AI 只做可选解释，不参与事实计算

当前系统运行时不需要 AI。outputs/metrics_matrix.csv 的 notes 是代码按分支生成的技术说明，不是 AI 生成内容。

如果未来确实需要 Claude API，唯一推荐场景是生成面向用户的摘要、证据导读或异常初步归类，并写入单独的 silver.ai_annotation：记录 model_id、prompt_version、input_hash、output_text、review_status 和 UTC 时间。AI 失败只能导致“没有注释”，不得阻塞确定性指标发布，更不能修改 metric_snapshot。

推荐落地顺序

  1. 先搭骨架。 建立 Unity Catalog、三个 schema、原始证据 Volume、权限组和一条 Lakeflow Job。
  2. 最小迁移现有程序。 将当前脚本打包为 Python wheel task；第一阶段仍可在 run-specific Volume 目录生成 CSV，再由 publish task 转成 Delta，不重写提取逻辑。
  3. 先完成版本发布。 实现 pipeline_run、dataset_version、metric_snapshot、metric_evidence 和 PUBLISHED 协议，导入当前结果作为首个版本。
  4. 再提供只读 API。 先实现 current、version、company metrics、evidence、history 五类接口，前端按版本缓存。
  5. 最后补季度能力。 当前主链路是 latest 10-K 年度口径；只有产品确认季度定义后，再加入 10-Q/10-Q/A、单季与 YTD 区分以及季度 Golden cases。

评审会上需要定下的四个问题

  • 网站是纯内网低并发，还是未来需要外网与严格亚秒 SLA？这决定是否需要 Lakebase，但不影响 Delta 主模型。
  • 产品默认展示“最新发布值”还是允许用户切换“截至某日”？建议两者都支持，首页默认最新。
  • 指标定义变更时，历史是否按旧定义保留？建议保留，并通过 definition_version 区分，绝不静默重算覆盖。
  • 第一阶段是否只上线当前年度指标？建议是；季度指标另开清晰的口径验收，不与平台迁移混在一次上线。

假设与边界

  • 本方案假设 Databricks 工作区已启用 Unity Catalog，并允许访问 SEC 官方域名；具体网络策略名称会因 AWS、Azure 或 GCP 而异。
  • 当前仓库的生产基线仍是年度 latest 10-K/10-K/A 口径；文中的 2026Q1 仅是版本模型示例，不代表季度提取已经实现。
  • Delta 的主键、外键和唯一约束不应被当作全部强制校验；发布任务仍需对业务键重复、缺失证据、非法状态和数值格式 Fail Fast。
  • 每个时间字段统一使用 UTC；前端仅在展示层转换用户时区。
